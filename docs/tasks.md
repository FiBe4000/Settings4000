## 2. System boundary

- [x] **2.1 `CommandRunner` trait + real impl** ŌĆö Complexity: 2
  Shell-free `Command` spawning (arg vectors), 5 s timeout, captured exit status/stderr; every invocation logged (cmd + exit, R7.3). Mock recorder impl for tests. *Accept:* unit tests via mock; real impl runs `true`/`false` correctly with timeout kill verified.
  *Done:* implemented in `src/system/command.rs`:
  - A `CommandRunner` trait over a shell-free `Command` (program + arg-vector builder ŌĆö no `sh`, no string interpolation) and a real `SystemCommandRunner`. `run` returns `Ok(CommandOutput)` (captured exit code + full stdout/stderr; `success()` == exit 0) for a completed process, and a `CommandError` distinguishing spawn failure, timeout, and wait failure. Both pipes are drained on per-stream reader threads concurrently with the timed wait, so a large-output command (e.g. `pw-dump` JSON, well over the 64 KiB pipe buffer) cannot block on write and deadlock. Each invocation and its exit status is logged at `info` (R7.3) with a truncated stderr excerpt only at `debug`.
  - The 5 s deadline is the named `DEFAULT_TIMEOUT` constant (architecture ┬¦6); a child that overruns it is killed and surfaces `CommandError::Timeout`. The wait-with-deadline uses the new `wait-timeout` = "0.2.1" dependency (race-free OS wait, no busy-poll). Tests inject a shorter timeout to verify the kill quickly while production stays at 5 s.
  - A `#[cfg(test)]` `MockCommandRunner` records the exact program+args sequence and returns queued canned outcomes (defaulting to success), so callers can be tested without spawning processes (R6.1).

- [x] **2.2 Atomic file writer** ŌĆö Complexity: 3
  Files are addressed by their XDG runtime path (`~/.config/ŌĆ”`), never a hardcoded `~/.dotfiles` path (R8.5). `fs::canonicalize` the target (a symlink into a dotfiles repo resolves to the repo file; a plain file resolves to itself) ŌåÆ `NamedTempFile` in the resolved target's dir ŌåÆ fsync ŌåÆ rename over the target (R5.4). Symlink preserved when present; plain files rewritten in place. Pre-write snapshot capture for rollback; write logged with resolved path (byte-level ŌĆö changed-key logging lives in the apply orchestrator, R7.3). *Accept:* tests prove symlink target rewritten + link preserved, a plain (non-symlink) file rewritten in place, no partial file on injected failure, snapshot restore works.
  *Done:* implemented in `src/system/writer.rs`:
  - `write_atomic(path, bytes)` canonicalizes the live XDG path (R8.5), snapshots the current bytes, then stages the new content in a `tempfile::NamedTempFile` in the resolved target's directory, fsyncs it, and atomically renames it over the resolved target (plus a best-effort parent-dir fsync). Renaming over the *resolved* path rewrites the real file and leaves any symlink into a dotfiles repo intact; a plain file is rewritten in place (R5.4). Original file permissions are preserved so the swap doesn't tighten a 0644 config to the temp file's 0600 (R8.3).
  - The content step is a closure seam (`atomic_write_with`), so a test can inject a failure before the rename and prove the original is untouched with no stray temp file (the `NamedTempFile` deletes on drop). `write_atomic` returns a `FileSnapshot` capturing the original bytes *and* permission mode, whose `restore()` puts both back through the same atomic path ŌĆö the per-file rollback the Apply pipeline (task 4.5) will use. Each write logs its resolved path + byte count at `info`, never contents (R7.3).
  - `tempfile` moved from a dev-dependency to a regular dependency because `NamedTempFile` is now used in production code (still available to tests).

- [x] **2.3 File freshness tracking** ŌĆö Complexity: 2
  Per-file content-hash + mtime record on read; `check_conflicts()` re-reads and reports changed files (R5.6). *Accept:* external modification between read and check is detected; unchanged files aren't flagged.
  *Done:* implemented in `src/core/freshness.rs` (domain logic, GTK-free ŌĆö architecture ┬¦3 "Write safety" and ┬¦6 step 2):
  - `FreshnessTracker::record(path)` reads a file and stores a per-file baseline of a content hash (`std`'s SipHash `DefaultHasher`, so no new dependency) plus its mtime; `record_bytes(path, bytes)` records the same baseline from bytes the caller already read (the store's read-time entry point, task 4.2) ŌĆö avoiding a second read and closing the TOCTOU gap where an external edit between the store's read and a tracker re-read would baseline the edited bytes; `check_conflicts()` re-reads every tracked file and returns a path-sorted `Vec<Conflict>` (each a tracked path + reason) of those that changed since it was recorded (R5.6). It lives in `core/` ŌĆö not `system/` ŌĆö because it is store/apply-pipeline domain logic (tasks 4.2/4.5) and reading files to compare them needs no side-effect abstraction.
  - Conflict rule: the content hash is authoritative ŌĆö a file conflicts iff its bytes no longer hash to the recorded value (`ContentChanged`) or it can no longer be re-read (`Unreadable`, e.g. deleted or permission-revoked, carrying the OS error); the recorded mtime is diagnostic only and never gates the decision, so a content edit made under an unchanged mtime is still caught and a pure mtime touch (or a rewrite to identical bytes) is not a false positive.

## 3. Parsers (each: lossless representation, targeted value edits, round-trip tests ŌĆö R5.3 item 1, R6.1)

- [x] **3.1 Palette kv parser (`colors/<scheme>`)** ŌĆö Complexity: 2
  Line-oriented `key=barehex`; in-place value edit; validation against the fixed 17-key schema. Write path exists only for R6.1 round-trip coverage / future per-key editing ŌĆö no v1 UI edits `colors/<scheme>` (R3.2 switching runs `generate-colors` only; per-key palette editing is out of scope for v1). *Accept:* round-trip identity; edit changes exactly one value span; malformed lines surfaced as parse warnings, not panics.
  *Done:* implemented in `src/parsers/palette.rs` (GTK-free parser, architecture ┬¦3):
  - Lossless representation: the file is kept as its lines in order, each retaining its exact raw bytes (terminator included) so `PaletteFile::emit()` concatenates them back byte-for-byte; a `key=value` entry additionally records the byte span of its value within that line. `parse()` never fails and preserves every line verbatim.
  - Edit API: `set_value(key, value)` splices only the recorded value span of the first matching entry, leaving indentation, spacing around `=`, trailing whitespace, comments, and all other lines byte-identical; it rejects a non-bare-hex value (`InvalidValue`, R8.3) or an absent/comment-only key (`UnknownKey`) before touching any byte. `value(key)` reads a key's current value.
  - Schema validation: `validate()` checks entry keys against the fixed 17-key schema (`PALETTE_KEYS`), reporting out-of-schema, missing, and duplicate keys; valid only when all three are empty (mirrors `generate-colors`' key-presence check, analysis ┬¦6.4). Value-format is a separate concern from key-presence.
  - Warning handling: malformed lines (no `key=value` shape) and non-hex values are returned as `ParseWarning`s (line number + kind) and logged at `warn` ŌĆö never panics, never data loss; comments/commented-out entries are never matched by edits or counted as entries.

- [x] **3.2 hyprlang parser/writer** ŌĆö Complexity: 5
  Tokenize comments, `key = value`, nested `section { }`, `source=`. Edits address keys by section path (`input.touchpad.natural_scroll`); repeatable top-level keys (`env=`, `exec-once=`) addressed by key + first comma-field (e.g. `env:XCURSOR_THEME`) with only the value portion edited; commented-out lines never matched; new keys appended at section end. Covers `config/hypr/input.conf` (the extracted `input { }` block), `hyprland.conf` (cursor `env=`), `hypridle.conf`, `hyprlock.conf`, `hyprpaper.conf`. *Accept:* round-trip identity on live-dotfile fixtures; targeted edits leave all other bytes untouched; nested + duplicate-section fixtures pass; the `input.*` fixture edits `input.conf` (e.g. `input.touchpad.natural_scroll`), not `hyprland.conf`; repeatable-key fixture (multiple `env=` lines, edit `env:XCURSOR_THEME` only) passes.
  *Done:* implemented in `src/parsers/hyprlang.rs` (GTK-free parser, architecture ┬¦3):
  - Token model: the file is kept as its lines in order, each retaining its exact raw bytes (terminator included) so `HyprlangFile::emit()` reproduces the input byte-for-byte; each line is classified as blank, comment, `section {` open, `}` close, `key = value` entry (recording the value's byte span, which stops before any trailing inline `# comment`), or malformed. An entry's key is the whitespace-free token before the first `=`, so hyprland's dotted keys (`col.active_border`) and `$variable` declarations classify correctly. Malformed lines and a `}`/header imbalance are surfaced as `ParseWarning`s and logged at `warn`, never panicking or losing data.
  - Section-path addressing (`KeyPath`): a value is named by a section path plus key; duplicate sections are told apart by 0-based occurrence (`SectionStep::nth`, e.g. hypridle's `listener[2]`), resolved by walking the open-section stack with per-parent occurrence counts so nesting and duplicates compose, and `section_count` enumerates repeated blocks (task 6.8's listeners). `input.*` paths resolve in `input.conf`, not `hyprland.conf`.
  - Repeatable-key addressing (`set_repeatable_field_value`): a top-level key (`env`, `source`, `exec-once`) is selected by its first comma-field and only the value portion after that comma is rewritten (e.g. `env:XCURSOR_THEME` ŌåÆ the cursor theme), leaving the field and siblings byte-identical; commented-out lines never match.
  - Append: `set_value` on a missing key inserts a new assignment just before its target section's closing `}` (or at end-of-file for a top-level key), copying an existing sibling's indentation and `=`-separator style, or falling back to the header indent + four spaces and ` = `. Values containing a newline or `#` are rejected before any byte changes (R8.3). Covered by round-trip identity on `input.conf`/`hypridle.conf`/`hyprpaper.conf`/`hyprland.conf`/`hyprlock.conf` fixtures plus targeted-edit, nested-and-duplicate-section, repeatable-key, commented-out, append, and malformed-input tests.

Malformed/unexpected `monitor=` lines are preserved and surfaced as warnings (no `unwrap`/panic).

- [x] **4.2 SettingsStore (staging state machine)** ŌĆö Complexity: 3
  `original`/`staged` per setting; dirty computation, per-page dirty rollup, `stage()`, `reset()`, conflict-triggered reload of originals (R5.1, R5.6). Runtime-only settings bypass staging (R5.2). *Accept:* headless tests cover stageŌåÆdirtyŌåÆresetŌåÆclean, conflict reload, and bypass paths.
  *Done:* implemented in `src/core/store.rs` (GTK-free domain logic ŌĆö R6.2), building on `model` (task 4.1) and `freshness` (task 2.3):
  - Original/staged model: `SettingsStore` keeps, per file-backed `SettingId`, an `original` `Value` (as read from disk) and an optional `staged` `Value` (the pending edit); dirty = a staged value that differs from the original (a staged value *equal* to the original is not dirty, so re-selecting the current value never lights up Apply). `stage(id, value)` records the edit, `value(id)` returns the effective (staged-or-original) value, and `original(id)` the baseline.
