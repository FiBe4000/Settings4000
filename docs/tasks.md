Atomic, non-overlapping breakdown of `docs/architecture.md`. Requirement references (R…) point to `docs/requirements.md`.
Complexity: 1 = simple/repetitive … 5 = highly complex.

## 2. System boundary

- [x] **2.1 `CommandRunner` trait + real impl** — Complexity: 2
  Shell-free `Command` spawning (arg vectors), 5 s timeout, captured exit status/stderr; every invocation logged (cmd + exit, R7.3). Mock recorder impl for tests. *Accept:* unit tests via mock; real impl runs `true`/`false` correctly with timeout kill verified.
  *Done:* implemented in `src/system/command.rs`:
  - A `CommandRunner` trait over a shell-free `Command` (program + arg-vector builder — no `sh`, no string interpolation) and a real `SystemCommandRunner`. `run` returns `Ok(CommandOutput)` (captured exit code + full stdout/stderr; `success()` == exit 0) for a completed process, and a `CommandError` distinguishing spawn failure, timeout, and wait failure. Both pipes are drained on per-stream reader threads concurrently with the timed wait, so a large-output command (e.g. `pw-dump` JSON, well over the 64 KiB pipe buffer) cannot block on write and deadlock. Each invocation and its exit status is logged at `info` (R7.3) with a truncated stderr excerpt only at `debug`.
  - The 5 s deadline is the named `DEFAULT_TIMEOUT` constant (architecture §6); a child that overruns it is killed and surfaces `CommandError::Timeout`. The wait-with-deadline uses the new `wait-timeout` = "0.2.1" dependency (race-free OS wait, no busy-poll). Tests inject a shorter timeout to verify the kill quickly while production stays at 5 s.
  - A `#[cfg(test)]` `MockCommandRunner` records the exact program+args sequence and returns queued canned outcomes (defaulting to success), so callers can be tested without spawning processes (R6.1).

- [x] **2.2 Atomic file writer** — Complexity: 3
  Files are addressed by their XDG runtime path (`~/.config/…`), never a hardcoded `~/.dotfiles` path (R8.5). `fs::canonicalize` the target (a symlink into a dotfiles repo resolves to the repo file; a plain file resolves to itself) → `NamedTempFile` in the resolved target's dir → fsync → rename over the target (R5.4). Symlink preserved when present; plain files rewritten in place. Pre-write snapshot capture for rollback; write logged with resolved path (byte-level — changed-key logging lives in the apply orchestrator, R7.3). *Accept:* tests prove symlink target rewritten + link preserved, a plain (non-symlink) file rewritten in place, no partial file on injected failure, snapshot restore works.
  *Done:* implemented in `src/system/writer.rs`:
  - `write_atomic(path, bytes)` canonicalizes the live XDG path (R8.5), snapshots the current bytes, then stages the new content in a `tempfile::NamedTempFile` in the resolved target's directory, fsyncs it, and atomically renames it over the resolved target (plus a best-effort parent-dir fsync). Renaming over the *resolved* path rewrites the real file and leaves any symlink into a dotfiles repo intact; a plain file is rewritten in place (R5.4). Original file permissions are preserved so the swap doesn't tighten a 0644 config to the temp file's 0600 (R8.3).
  - The content step is a closure seam (`atomic_write_with`), so a test can inject a failure before the rename and prove the original is untouched with no stray temp file (the `NamedTempFile` deletes on drop). `write_atomic` returns a `FileSnapshot` capturing the original bytes *and* permission mode, whose `restore()` puts both back through the same atomic path — the per-file rollback the Apply pipeline (task 4.5) will use. Each write logs its resolved path + byte count at `info`, never contents (R7.3).
  - `tempfile` moved from a dev-dependency to a regular dependency because `NamedTempFile` is now used in production code (still available to tests).

- [x] **2.3 File freshness tracking** — Complexity: 2
  Per-file content-hash + mtime record on read; `check_conflicts()` re-reads and reports changed files (R5.6). *Accept:* external modification between read and check is detected; unchanged files aren't flagged.
  *Done:* implemented in `src/core/freshness.rs` (domain logic, GTK-free — architecture §3 "Write safety" and §6 step 2):
  - `FreshnessTracker::record(path)` reads a file and stores a per-file baseline of a content hash (`std`'s SipHash `DefaultHasher`, so no new dependency) plus its mtime; `record_bytes(path, bytes)` records the same baseline from bytes the caller already read (the store's read-time entry point, task 4.2) — avoiding a second read and closing the TOCTOU gap where an external edit between the store's read and a tracker re-read would baseline the edited bytes; `check_conflicts()` re-reads every tracked file and returns a path-sorted `Vec<Conflict>` (each a tracked path + reason) of those that changed since it was recorded (R5.6). It lives in `core/` — not `system/` — because it is store/apply-pipeline domain logic (tasks 4.2/4.5) and reading files to compare them needs no side-effect abstraction.
  - Conflict rule: the content hash is authoritative — a file conflicts iff its bytes no longer hash to the recorded value (`ContentChanged`) or it can no longer be re-read (`Unreadable`, e.g. deleted or permission-revoked, carrying the OS error); the recorded mtime is diagnostic only and never gates the decision, so a content edit made under an unchanged mtime is still caught and a pure mtime touch (or a rewrite to identical bytes) is not a false positive.

## 3. Parsers (each: lossless representation, targeted value edits, round-trip tests — R5.3 item 1, R6.1)

- [x] **3.1 Palette kv parser (`colors/<scheme>`)** — Complexity: 2
  Line-oriented `key=barehex`; in-place value edit; validation against the fixed 17-key schema. Write path exists only for R6.1 round-trip coverage / future per-key editing — no v1 UI edits `colors/<scheme>` (R3.2 switching runs `generate-colors` only; per-key palette editing is out of scope for v1). *Accept:* round-trip identity; edit changes exactly one value span; malformed lines surfaced as parse warnings, not panics.
  *Done:* implemented in `src/parsers/palette.rs` (GTK-free parser, architecture §3):
  - Lossless representation: the file is kept as its lines in order, each retaining its exact raw bytes (terminator included) so `PaletteFile::emit()` concatenates them back byte-for-byte; a `key=value` entry additionally records the byte span of its value within that line. `parse()` never fails and preserves every line verbatim.
  - Edit API: `set_value(key, value)` splices only the recorded value span of the first matching entry, leaving indentation, spacing around `=`, trailing whitespace, comments, and all other lines byte-identical; it rejects a non-bare-hex value (`InvalidValue`, R8.3) or an absent/comment-only key (`UnknownKey`) before touching any byte. `value(key)` reads a key's current value.
  - Schema validation: `validate()` checks entry keys against the fixed 17-key schema (`PALETTE_KEYS`), reporting out-of-schema, missing, and duplicate keys; valid only when all three are empty (mirrors `generate-colors`' key-presence check, analysis §6.4). Value-format is a separate concern from key-presence.
  - Warning handling: malformed lines (no `key=value` shape) and non-hex values are returned as `ParseWarning`s (line number + kind) and logged at `warn` — never panics, never data loss; comments/commented-out entries are never matched by edits or counted as entries.

- [x] **3.2 hyprlang parser/writer** — Complexity: 5
  Tokenize comments, `key = value`, nested `section { }`, `source=`. Edits address keys by section path (`input.touchpad.natural_scroll`); repeatable top-level keys (`env=`, `exec-once=`) addressed by key + first comma-field (e.g. `env:XCURSOR_THEME`) with only the value portion edited; commented-out lines never matched; new keys appended at section end. Covers `config/hypr/input.conf` (the extracted `input { }` block), `hyprland.conf` (cursor `env=`), `hypridle.conf`, `hyprlock.conf`, `hyprpaper.conf`. *Accept:* round-trip identity on live-dotfile fixtures; targeted edits leave all other bytes untouched; nested + duplicate-section fixtures pass; the `input.*` fixture edits `input.conf` (e.g. `input.touchpad.natural_scroll`), not `hyprland.conf`; repeatable-key fixture (multiple `env=` lines, edit `env:XCURSOR_THEME` only) passes.
  *Done:* implemented in `src/parsers/hyprlang.rs` (GTK-free parser, architecture §3):
  - Token model: the file is kept as its lines in order, each retaining its exact raw bytes (terminator included) so `HyprlangFile::emit()` reproduces the input byte-for-byte; each line is classified as blank, comment, `section {` open, `}` close, `key = value` entry (recording the value's byte span, which stops before any trailing inline `# comment`), or malformed. An entry's key is the whitespace-free token before the first `=`, so hyprland's dotted keys (`col.active_border`) and `$variable` declarations classify correctly. Malformed lines and a `}`/header imbalance are surfaced as `ParseWarning`s and logged at `warn`, never panicking or losing data.
  - Section-path addressing (`KeyPath`): a value is named by a section path plus key; duplicate sections are told apart by 0-based occurrence (`SectionStep::nth`, e.g. hypridle's `listener[2]`), resolved by walking the open-section stack with per-parent occurrence counts so nesting and duplicates compose, and `section_count` enumerates repeated blocks (task 6.8's listeners). `input.*` paths resolve in `input.conf`, not `hyprland.conf`.
  - Repeatable-key addressing (`set_repeatable_field_value`): a top-level key (`env`, `source`, `exec-once`) is selected by its first comma-field and only the value portion after that comma is rewritten (e.g. `env:XCURSOR_THEME` → the cursor theme), leaving the field and siblings byte-identical; commented-out lines never match.
  - Append: `set_value` on a missing key inserts a new assignment just before its target section's closing `}` (or at end-of-file for a top-level key), copying an existing sibling's indentation and `=`-separator style, or falling back to the header indent + four spaces and ` = `. Values containing a newline or `#` are rejected before any byte changes (R8.3). Covered by round-trip identity on `input.conf`/`hypridle.conf`/`hyprpaper.conf`/`hyprland.conf`/`hyprlock.conf` fixtures plus targeted-edit, nested-and-duplicate-section, repeatable-key, commented-out, append, and malformed-input tests.

- [x] **3.3 monitors.conf record parser** — Complexity: 3
  Parse `monitor=` records (name/desc, mode, position, scale, enable — the latter handled as `monitor=NAME,disable` records) preserving surrounding lines and later-rule-wins order; edit matching record or append after last `monitor=` line. *Accept:* round-trip identity; editing the eDP rule doesn't disturb the catchall or comments.
  *Done:* implemented in `src/parsers/monitors.rs` (GTK-free parser, architecture §3):
  - Lossless line model like the sibling parsers: each `monitor=` record keeps its raw bytes plus the trimmed byte span of every comma-separated field (index 0 = name, 1 = mode, 2 = position, 3 = scale, 4.. = extras) and a `disabled` flag for the `NAME,disable` form; `emit()` reproduces the input byte-for-byte and records are never reordered, so Hyprland's later-rule-wins order is preserved.
  - `set_field` rewrites exactly one field span (mode/position/scale) in place, and `set_state` enables/disables a record or **appends** a new one after the last `monitor=` line (copying the file's `monitor=` separator style); both reject a comma/newline/`#` in a written value so the record stays parseable by the awk in `scripts/hypr-display-profile.sh` (name field 1, mode field 2, scale field 4) — the app never touches that script.
  - Built as a focused standalone parser rather than on the hyprlang line model: hyprlang addresses a repeatable key by its first comma-field and edits the rest as one blob, whereas this needs positional field addressing (match by name, edit arbitrary fields) specific to `monitors.conf`. Malformed/unexpected `monitor=` lines are preserved and surfaced as warnings (no `unwrap`/panic).

- [x] **3.4 swaync JSON adapter** — Complexity: 1
  `serde_json` preserve_order, parse–modify–reserialize (2-space pretty). *Accept:* key order stable across a no-op edit; position/timeout edits round-trip.
  *Done:* implemented in `src/parsers/swaync.rs` (GTK-free adapter, architecture §3):
  - `SwayncConfigFile::parse` decodes `config.json` into a `serde_json` object map (the crate's `preserve_order` feature backs objects with an index map, so key order is retained); `emit()` re-serializes with the 2-space pretty printer plus a trailing newline — swaync's exact on-disk shape, verified by a byte-identity round-trip test. Malformed JSON returns `ParseError::InvalidJson` and a non-object root `ParseError::NotAnObject`; neither panics.
  - Get/set API is by key with typed access for the Notifications page (task 6.7): `string`/`set_string` (position enums `positionX`/`positionY`), `integer`/`set_integer` (`timeout`/`timeout-low`/`timeout-critical`), and `boolean`/`set_boolean` (top-level flags such as `keyboard-shortcuts`/`fit-to-screen`). Getters return `None` for an absent or wrongly-typed key; setters update in place (preserving key order) or append a new key. Domain validation (enum/range) is intentionally left to task 4.1/6.7. Note: swaync's do-not-disturb is runtime daemon state (via `swaync-client`), not a config boolean, so task 6.7 must confirm the DND mechanism rather than write a `dnd` key.
  - Formatting caveat: JSON is normalized, not byte-preserved — a file already in 2-space-pretty-plus-newline form round-trips exactly, while other formatting (different indent, no trailing newline) is rewritten to the canonical form. This is the only unavoidable difference and is expected for a comment-free JSON file.

- [x] **3.5 INI editor (GTK settings.ini)** — Complexity: 2
  Line-preserving kv-in-section edits. Both `settings.ini` files are now tracked in the dotfiles and symlinked (analysis §6.5), so editing an existing `[Settings]` section is the common path; creating file + `[Settings]` section when absent remains a fallback. *Accept:* comments/order preserved; both the edit-existing and file-creation paths tested.
  *Done:* implemented in `src/parsers/ini.rs` (GTK-free editor, architecture §3):
  - Lossless line model like the sibling parsers: each line keeps its raw bytes (terminator included) and a classification (blank / `#` comment / `[section]` header / `key=value` entry with the value's byte span / malformed), so `IniFile::emit()` reproduces the input byte-for-byte and a malformed line (e.g. an unclosed `[`) is preserved and surfaced as a `ParseWarning`, never a panic.
  - `set_value(section, key, value)` unifies all three paths, reporting which happened via `SetOutcome`: **edit-existing** (the common case on the dotfiles machine) splices only the value span inside an existing `[Settings]`; **append** inserts an absent key at the section's end (after its last non-blank line, copying a sibling entry's indent/separator style); **create-from-scratch** builds `[Settings]` + the key at EOF (from `IniFile::empty()` or a file lacking the section). Values with a newline/carriage return, and unusable keys/section names, are rejected before any byte changes (R8.3).
  - Per-file by design: the Theme page (task 6.4) drives both `gtk-3.0`/`gtk-4.0` files by applying identical `set_value` calls to each `IniFile` (covered by a test). Caveat: this produces the *bytes* of a created file only — physically creating an absent `settings.ini` on disk is a writer/Apply concern (the atomic writer only rewrites existing targets today), deferred to tasks 4.5/6.4.

- [x] **3.6 env-file editor (`uwsm/env`)** — Complexity: 2
  Targeted `export KEY=value` edits for `XCURSOR_THEME`/`XCURSOR_SIZE`; detects (but never edits) commented lines — feeds the `GTK_THEME` override warning (R3.3). *Accept:* round-trip identity; commented `GTK_THEME` reported, uncommented lines editable.
  *Done:* implemented in `src/parsers/env.rs` (GTK-free editor, architecture §3):
  - Lossless line model like the sibling parsers: each line keeps its raw bytes (terminator included) and a classification (blank / `#` comment / `KEY=value` assignment with the value's byte span / malformed), so `EnvFile::emit()` reproduces the input byte-for-byte and only a genuinely uninterpretable line (no `=`) is surfaced as a `ParseWarning`, never a panic. Because uwsm sources the file with allexport, both `export KEY=value` and a bare `KEY=value` are recognized as assignments.
  - `set_value(key, value)` rewrites only the value span of the **last** (shell-effective) assignment for the key (either form), or appends `export KEY=value` at EOF when the key is absent — value-only so the Theme page (task 6.4) can drive this copy and the other `XCURSOR_*` copies from one value. Quotes are treated as part of the value span (never parsed/synthesized); a written value with a newline/carriage return is rejected (R8.3).
  - `gtk_theme_override()` reads (never edits) `GTK_THEME` for R3.3, distinguishing an active `GTK_THEME=…` (export or bare → banner + disabled drop-down) from a commented-out one (`#export …`, `# export …`, or `# GTK_THEME=…`, present but inactive) and absent; a commented line is never matched by a value-set call.

- [x] **3.7 Generated-file readers** — Complexity: 1
  Read-only: active-scheme detection from the `# Generated from colors/<scheme>` header of the deployed `~/.config/hypr/colors.conf` (R3.2); optional per-scheme swatch parse (reads the discovered repo's `colors/<scheme>` files, R8.5) for the drop-down. The repo also carries a `state/active-scheme` marker (analysis §6.4) as an optional future detection shortcut; v1 reads the header. *Accept:* scheme name extracted from fixture; missing/odd header degrades to "unknown" without error.
  *Done:* implemented in `src/parsers/generated.rs` (GTK-free, read-only — the only parser module with no write/edit path, since the generated files must never be written by the app):
  - Active-scheme detection reads the `# Generated from colors/<scheme>` header of the deployed generated file (v1: `~/.config/hypr/colors.conf`) and returns a typed `ActiveScheme::Named(name)` or `ActiveScheme::Unknown` (R3.2). Recognition is tolerant (leading whitespace, optional space after `#`, collapsed internal whitespace, ASCII-case-insensitive prose and `colors/` prefix) but does not over-match; a missing header, an odd/malformed header, an empty file, or an unreadable file all degrade to `Unknown` — never an `Err`/panic. The `state/active-scheme` marker is noted as an optional future shortcut but v1 reads the header.
  - Per-scheme swatch parse (`parse_scheme_swatch`/`read_scheme_swatch`, R8.5) reuses the task-3.1 palette parser to read the 17 schema colors (in canonical order) plus the schema-validity report into a read-only `SchemeSwatch` for the theme drop-down (task 6.3); it degrades gracefully on a malformed palette (fewer colors + a validation report, no panic). Callers pass file contents or a live path (repo-root discovery is task 4.3).

## 4. Core domain (GTK-free — R6.2)

- [x] **4.1 Typed settings model + validators** — Complexity: 3
  `SettingId`, typed values (enum/bool/float/string), per-setting validation: hex colors, `WxH@Hz` mode strings, timeout/sensitivity/scale ranges, file paths (exists + readable + image extension for wallpaper/lock background) (R8.3). *Accept:* unit tests per validator, valid + invalid cases.
  *Done:* implemented in `src/core/model.rs` (GTK-free domain logic — R6.2):
  - `SettingId` is a fieldless enum — the central registry of every editable setting, used as the store's key (task 4.2) and validated by the Apply pipeline (task 4.5). Each variant declares its `kind()` (a `ValueKind`) and `category()` (its sidebar page, for the store's per-page dirty rollup); a page adds a setting by adding a variant and extending those mappings plus the `validate` match (task 4.2 later added a `backing()` mapping — file-backed vs runtime-only — alongside `kind()`/`category()`/`validate()`). `Value` is a typed union — `Enum`/`Bool`/`Float`/`Integer`/`String` with typed accessors — where `Integer` is kept distinct from `Float` so whole-number settings (timeouts) never gain a fractional part or emit `300.0` into a config expecting `300`.
  - `SettingId::validate(&Value)` first requires the value's kind to match (`WrongKind`), then dispatches to the value-format validator; the returned `ValidationError` renders a UI-ready message via `Display` (R8.3). Validators: bare six-hex-digit color (reuses the task-3.1 `is_bare_hex`); monitor mode accepting a bare `WxH`, a full `WxH@Hz`, or the `preferred`/`highres`/`highrr` tokens (dimensions 1–16384 px, and when present a 1–1000 Hz refresh); numeric ranges `SENSITIVITY_RANGE` (-1.0..=1.0), `SCALE_RANGE` (0.5..=3.0), `TIMEOUT_SECONDS_RANGE` (1..=86400 s); and an image-path check (exists + regular file + png/jpg/jpeg/webp extension + readable, following symlinks, no panic on an IO error).
  - Unit-tested per validator with valid and invalid cases (hex length/chars/`#`-prefix; malformed and out-of-range modes; range boundaries and just-outside; image path missing/directory/non-image/unreadable via a `tempfile` fixture), plus dispatch, wrong-kind, and `Display`-message coverage; the module-boundary guard confirms `core/` stays GTK-free.

- [x] **4.2 SettingsStore (staging state machine)** — Complexity: 3
  `original`/`staged` per setting; dirty computation, per-page dirty rollup, `stage()`, `reset()`, conflict-triggered reload of originals (R5.1, R5.6). Runtime-only settings bypass staging (R5.2). *Accept:* headless tests cover stage→dirty→reset→clean, conflict reload, and bypass paths.
  *Done:* implemented in `src/core/store.rs` (GTK-free domain logic — R6.2), building on `model` (task 4.1) and `freshness` (task 2.3):
  - Original/staged model: `SettingsStore` keeps, per file-backed `SettingId`, an `original` `Value` (as read from disk) and an optional `staged` `Value` (the pending edit); dirty = a staged value that differs from the original (a staged value *equal* to the original is not dirty, so re-selecting the current value never lights up Apply). `stage(id, value)` records the edit, `value(id)` returns the effective (staged-or-original) value, and `original(id)` the baseline.
  - Dirty + rollup: `is_dirty()` drives the global Apply state, `is_category_dirty(Category)` is the per-page modified-marker rollup, and `dirty_ids()` lists the changed settings in `SettingId` order for the Apply pipeline (task 4.5). `reset()` discards every staged edit, returning the store to clean.
  - Conflict-reload (R5.6): the store owns a `FreshnessTracker` and baselines each file via `record_bytes(path, &bytes)` — the exact bytes the loader parsed the original from, never a re-read (avoids the double-read + the read→baseline TOCTOU window per the 2.3 review). `refresh()` re-reads the tracked files, and for each that changed externally reloads its originals from disk through a per-file reader closure (keeping the store parser-agnostic) and re-baselines; pending staged edits are preserved (no silent clobber), and the `RefreshReport` tells the UI which files reloaded vs. failed (deleted/unreadable).
  - Runtime-only bypass (R5.2): a new `Backing` marker on `SettingId` (`SettingId::backing()`) classifies each setting; `stage()` returns `StageOutcome::RuntimeBypass` for a runtime-only setting (the laptop-display toggle, added as the first such variant) without storing anything, so it is never staged and never dirty. The store's map holds only file-backed settings, so dirty/rollup/reset exclude runtime ones automatically.
  - Dirty-robustness (the 4.1 review, `Value` is `PartialEq` not `Eq`): `stage()` validates every value via `SettingId::validate` first, and the range validators reject a non-finite float, so a `NaN` can never enter `staged` and cause a permanently-dirty setting; `reset()` is independently safe because it clears `staged` to `None` (dirty is false without a staged value).
  - Tested headless: stage→dirty→reset→clean, per-page rollup, external-change conflict reload (tempfile edited between load and refresh) with re-baselining and staged-edit preservation, the runtime-only bypass, and the NaN/out-of-range/unloaded validate-on-stage cases.

- [x] **4.3 Capabilities detection** — Complexity: 3
